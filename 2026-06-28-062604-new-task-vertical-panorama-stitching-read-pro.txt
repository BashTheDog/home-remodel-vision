  2. It returns one or more bounding boxes with confidence scores.
  3. The script picks the highest-confidence box.
  4. That box is handed to SAM2, which traces the exact object boundary inside
  it.

  Tips:
  - Use a simple noun: "couch", "lamp", "window", "chair", "rug". Multi-word
  phrases work too: "potted plant", "floor lamp".
  - If nothing is found, try a more general word (e.g. "furniture" instead of
  "sectional sofa").
  - If the wrong object is picked (e.g. two couches and it grabs the smaller
  one), use Method B or C to be specific.
  - Confidence 0.38 is on the low end — if the result looks wrong, lower
  GDINO_BOX_THR from 0.35 to 0.25 in remove_object.py to catch weaker
  detections, or raise it to 0.50 to filter out false positives.

  ---
  Method B — Click point

  ./remove_object.sh <image> <name> --point 1200,800

  What happens:
  1. Skips GroundingDINO entirely.
  2. SAM2 receives a single (x, y) pixel coordinate as a positive click — "the
  object is here."
  3. SAM2 figures out what object fills that location and traces its boundary.

  How to find the right coordinates:
  - Open the image in any viewer (the pano viewer works — the zoom-badge shows
  100% scale, but the coordinates you need are in the original image's pixel 
  space, not screen pixels).
  - Easiest method: open the before image in macOS Preview → Tools → Show
  Inspector (⌘I) → hover over the object's center. The cursor position shown is
  the pixel coordinate you want.
  - Click somewhere clearly inside the object, not on its edge.
  - For a large object, click near its center mass.

  Example for a couch at roughly the middle of the right half of an 11K image:
  --point 9400,4800

  ---
  Method C — Rough bounding box

  ./remove_object.sh <image> <name> --box 7500,3700,11200,5800

  Format: x1,y1,x2,y2 — top-left corner, bottom-right corner, in original image
  pixels.

  What happens:
  1. Skips GroundingDINO.
  2. SAM2 receives the box as a spatial hint: "the object is somewhere inside
  this rectangle."
  3. SAM2 finds the dominant object within the box and traces it precisely.

  The box does NOT need to be tight. A loose box with 100–200px of padding
  around the object works fine — SAM2 handles the boundary. A box that clips the
  object will produce a clipped mask.

  How to get box coordinates:
  - In macOS Preview with the selection tool (⌘3), draw a rectangle around the
  object. The Inspector (⌘I) shows the selection's origin (x, y) and size (w,
  h). Convert: x1=x, y1=y, x2=x+w, y2=y+h.
  - Or use the pano viewer to pan/zoom to the object, then estimate from the
  image dimensions.

❯ describe this pipeline                                                        

● What's been built

  Two self-contained stages that run on the DGX Spark and are viewed from a Mac
  via SSH tunnel.

  ---
  Stage 1 — Vertical panorama stitch

  Takes two overlapping photos of the same scene (upper shot, lower shot) and
  produces one tall composite image.

  How it works:
  1. Tries cv2.Stitcher (OpenCV's built-in, flat projection) first.
  2. Falls back to SIFT feature detection at ¼ scale → FLANN matching → RANSAC
  homography → feathered vertical blend in the overlap zone.
  3. Outputs a single JPEG to outputs/pano/.

  Interface: stitch_pano.sh <upper> <lower> <name>

  Viewer: pan/zoom browser viewer on port 8081, scroll to zoom, drag to pan,
  double-click to reset fit.

  ---
  Stage 2 — Object removal

  Takes any photo (including the stitched panorama), removes a named or
  pointed-at object, and fills the hole with AI-generated background.

  Three steps in sequence:

  1. Selection — two modes:
  - Text prompt (--prompt "couch"): GroundingDINO searches the image for objects
  matching the description, returns bounding boxes with confidence scores.
  - Manual (--point x,y or --box x1,y1,x2,y2): skips detection entirely, goes
  straight to segmentation with user-supplied coordinates.

  2. Segmentation — SAM2 (large model) produces a precise pixel mask.
  Critically, it runs on a crop around the detected region rather than the full
  image — on an 11K-pixel panorama, feeding the full image would downsample the
  object 11× and produce coarse edges. The crop gives SAM2 full resolution on
  the target. The mask is then dilated slightly so LaMa has clean edges to work
  with.

  3. Inpainting — LaMa (a Fourier-convolution inpainting model) fills the masked
  hole. Again runs on a crop (mask bounding box + 200px margin) to avoid a
  PyTorch 32-bit index limit on very large images, then pastes back.

  Outputs: _before.jpg, _mask.jpg (red overlay), _after.jpg in outputs/edited/.

  Interface: remove_object.sh <image> <name> --prompt "couch"

  Viewer: before/after comparison viewer on port 8082 — draggable center divider
  to compare, plus separate Before / After / Mask buttons, with pan and zoom.

  ---
  Shared infrastructure

  - All models run entirely on the Spark (GroundingDINO, SAM2, LaMa — ~1.1GB of
  weights on disk).
  - Both viewers are plain Python HTTP servers bound to 0.0.0.0, reachable from
  the Mac via ssh -L <port>:<container-ip>:<port> spark-pharn.local. The
  container IP is printed automatically at startup.
  - No Node, no npm, no external services — just Python and a browser.

❯ what open source ai models are used in this pipeline                          

● Four models across the two stages:

  ---
  Stage 1 — Panorama stitch

  None. Pure OpenCV (SIFT feature detector, FLANN matcher, RANSAC homography).
  No neural network.

  ---
  Stage 2 — Object removal

  GroundingDINO (IDEA Research, 2023)
  Detects objects in an image from a free-text description. Architecture: Swin
  Transformer image backbone + BERT text encoder, fused with a transformer
  decoder. Returns bounding boxes and confidence scores.

  SAM2 — Segment Anything Model 2 (Meta, 2024)
  Takes a bounding box, point, or mask prompt and produces a precise pixel-level
  segmentation mask. Architecture: Hiera vision transformer backbone + mask
  decoder. We use the large variant (857MB).

  LaMa — Large Mask inpainting (Samsung Research, 2021)
  Fills the masked hole with plausible background. Architecture:
  fully-convolutional network with Fast Fourier Convolution blocks, which lets
  it reason about repeating textures (floors, walls, fabric) across large
  distances. Packaged here via simple-lama-inpainting.

  ---
  All three are Apache 2.0 licensed and run fully locally — no API calls, no
  cloud, no data leaves the Spark.

❯ what about Stable Diffusion WebUI with ControlNet                             

● Not in this pipeline. What's here is the minimal path to object removal —
  detect, mask, inpaint — using models that work on ARM64/sm_121 without a
  fight.

  Stable Diffusion WebUI + ControlNet would be a different approach to the
  inpainting step. Instead of LaMa's fast texture-fill, you'd get
  diffusion-based inpainting that can hallucinate plausible content (not just
  repeat surrounding texture) — better for removing large objects where the
  background is complex or unknown.

  Whether it's worth adding depends on what you're actually removing:

  ┌────────────────────────────────────┬───────────────────┬────────────────┐
  │              Scenario              │  LaMa (current)   │     SD +       │
  │                                    │                   │   ControlNet   │
  ├────────────────────────────────────┼───────────────────┼────────────────┤
  │ Floor, wall, plain background      │ Great — fast,     │ Overkill       │
  │                                    │ clean             │                │
  ├────────────────────────────────────┼───────────────────┼────────────────┤
  │ Large furniture against textured   │ Acceptable        │ Better         │
  │ wall                               │                   │                │
  ├────────────────────────────────────┼───────────────────┼────────────────┤
  │ Object with complex background     │ Often smears      │ Much better    │
  │ behind it                          │                   │                │
  ├────────────────────────────────────┼───────────────────┼────────────────┤
  │ Speed                              │ ~5s on the Spark  │ ~30–60s per    │
  │                                    │                   │ run            │
  └────────────────────────────────────┴───────────────────┴────────────────┘

  If the LaMa results look smeared or implausible on your actual photos, SD
  inpainting is the natural next upgrade. The ControlNet piece would let you
  condition the inpaint on the surrounding depth or structure, keeping
  perspective consistent — useful for room shots.

  It could be added as a flag to the existing script (--inpainter 
  lama|diffusion) rather than a separate stage, so you'd pick per-job. But
  that's Stage 3 territory — do you want to go that direction, or is there
  something else you had in mind for Stage 3?
